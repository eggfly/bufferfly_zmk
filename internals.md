# CyberFly 底层原理

## BLE Profile 机制

### 5 个 Profile 是 ZMK 的限制，不是硬件限制

ZMK 构建时 `CONFIG_BT_MAX_PAIRED=5`，这是 Zephyr 蓝牙协议栈的软件配置：

```
Zephyr 默认: CONFIG_BT_MAX_PAIRED=1 (最少 1，最大 128)
ZMK 覆盖为: CONFIG_BT_MAX_PAIRED=5
```

nRF52840 的 SoftDevice S140 硬件上支持最多 **20 个并发连接**。ZMK 限制为 5 是因为：
- 每个 Profile 需要存储配对密钥到 NVS (flash)，占存储空间
- 键盘场景下 5 个设备足够用
- 可以通过在 `.conf` 里改 `CONFIG_BT_MAX_PAIRED=10` 来增加

### Profile 工作方式

```
         Profile 0          Profile 1          Profile 2
         ┌─────────┐       ┌─────────┐       ┌─────────┐
         │ Android │       │  iOS    │       │  (空)   │
         │ 已配对   │       │ 已配对   │       │ 可配对   │
         └─────────┘       └─────────┘       └─────────┘
              ▲
              │
         当前活跃 Profile
```

- 键盘同一时刻只向**一个 Profile** 发送按键
- 通过 `&bt BT_SEL x` 切换活跃 Profile
- 通过 `&bt BT_NXT` / `&bt BT_PRV` 顺序切换
- 通过 `&bt BT_CLR` 清除当前 Profile 的配对
- 切换 Profile 后，键盘会自动断开旧设备、连接新设备

### 为什么配对失败

当 Profile 0 已绑定 Android 时：
1. iOS 尝试连接 → 走到 `auth_pairing_accept()` 回调
2. ZMK 检查 `zmk_ble_active_profile_is_open()` → Profile 0 已被占 → 返回 `BT_SECURITY_ERR_PAIR_NOT_ALLOWED`
3. iOS 收到拒绝 → 报错"无法配对"

**解决方案**: 先切到空的 Profile 1 (`&bt BT_SEL 1`)，再让 iOS 配对。

## nRF52840 vs ESP32-C3 蓝牙对比

### 都是 BLE，但架构完全不同

两者都支持 **Bluetooth Low Energy (BLE)**，符合 Bluetooth 4.0+ 规范。但硬件实现差异巨大：

| 特性 | nRF52840 | ESP32-C3 |
|------|----------|----------|
| CPU | ARM Cortex-M4F @ 64MHz | RISC-V 单核 @ 160MHz |
| 蓝牙版本 | Bluetooth 5.0 | Bluetooth 5.0 |
| BLE 支持 | 完整 (Long Range, 2Mbps, Coded PHY) | 基础 BLE |
| WiFi | 无 | WiFi 4 (802.11 b/g/n) |
| 射频架构 | **专用 BLE 射频** | **WiFi/BLE 共用射频** |
| Flash | 1MB 内置 | 4MB 外置 SPI Flash |
| RAM | 256KB | 400KB |
| 典型 BLE 功耗 | **~5mA 收发** | ~50-130mA 收发 |
| 深度睡眠 | **0.4μA** (System OFF) | ~5μA (Deep Sleep) |
| 浅睡眠 | **1.5μA** (System ON + RTC) | ~800μA (Light Sleep) |
| 供电范围 | 1.7-5.5V | 3.0-3.6V |
| 价格 | ~¥15-25 | ~¥5-10 |
| 适合场景 | **电池供电低功耗设备** | WiFi+BLE 联网设备 |

### 为什么 nRF52840 能做到超低功耗

#### 1. 专用射频硬件 (Dedicated Radio)

```
nRF52840:
┌──────────────────────────────┐
│  CPU (Cortex-M4F)            │  ← 大部分时间在睡觉
│  ┌────────────────────────┐  │
│  │ 2.4GHz Radio (专用BLE) │  │  ← 独立射频，只做 BLE
│  │ - 片上巴伦 (on-chip)   │  │
│  │ - 硬件 AES-128 加密    │  │
│  │ - 硬件地址匹配 (AAR)   │  │
│  └────────────────────────┘  │
└──────────────────────────────┘

ESP32-C3:
┌──────────────────────────────┐
│  CPU (RISC-V)                │  ← 需要更多 CPU 参与
│  ┌────────────────────────┐  │
│  │ 2.4GHz Radio           │  │  ← WiFi + BLE 共用射频
│  │ - 需要时分复用          │  │     同一时刻只能做一件事
│  │ - WiFi 天线匹配更复杂   │  │
│  └────────────────────────┘  │
└──────────────────────────────┘
```

nRF52840 的射频是专门为 BLE 优化的：
- **片上巴伦** (balun)：不需要外部 RF 匹配元件（实际还是建议加）
- **硬件加密**：AES-128/CCM 在硬件完成，CPU 不参与
- **硬件地址解析** (AAR)：识别已配对设备不需要 CPU

ESP32-C3 的射频同时服务 WiFi 和 BLE：
- WiFi 和 BLE 时分复用同一个射频前端
- WiFi 需要更高功率（~150mA 发射 vs BLE ~5mA）
- 射频前端的功率放大器 (PA) 为 WiFi 设计，BLE 时效率不是最优

#### 2. PPI 系统 (Programmable Peripheral Interconnect)

这是 nRF 系列最核心的省电机制：

```
传统 MCU (如 ESP32):
  定时器到期 → 中断 CPU → CPU 醒来 → CPU 读定时器 → CPU 启动 ADC → CPU 等待 → CPU 读结果
  (CPU 全程参与，全程耗电)

nRF52840 PPI:
  定时器到期 ──PPI通道──→ 自动启动 ADC ──PPI通道──→ 自动启动 DMA 传输
  (CPU 全程睡觉，0 参与)
```

PPI 允许外设之间**直接通信**，不经过 CPU：
- 定时器可以直接触发 GPIO 翻转
- 射频事件可以直接触发定时器
- GPIO 变化可以直接触发 ADC 采样
- 所有这些都在 CPU 睡眠时完成

nRF52840 有 **20 个 PPI 通道 + 12 个预编程通道**，BLE 协议栈大量使用 PPI：
- 射频收发的精确时序靠 PPI + Timer 完成
- CPU 只在需要处理数据包内容时才被唤醒
- 一次 BLE 连接事件中，CPU 可能只醒来几十微秒

#### 3. 多级电源管理

```
nRF52840 电源模式:

┌─────────────────────────────────────────────────────┐
│ 模式           │ 电流      │ 唤醒时间  │ 保持内容    │
├─────────────────────────────────────────────────────┤
│ Active (CPU跑) │ ~3-5mA    │ -        │ 全部       │
│ System ON Idle │ 1.5μA     │ <2μs     │ RAM + RTC  │
│ System OFF     │ 0.4μA     │ ~1ms     │ 可选 RAM   │
│ System OFF     │ 0.3μA     │ ~1ms     │ 无 RAM     │
└─────────────────────────────────────────────────────┘

ESP32-C3 电源模式:

┌─────────────────────────────────────────────────────┐
│ 模式           │ 电流      │ 唤醒时间  │ 保持内容    │
├─────────────────────────────────────────────────────┤
│ Active         │ ~50-130mA │ -        │ 全部       │
│ Modem Sleep    │ ~15mA     │ 即时     │ 全部       │
│ Light Sleep    │ ~800μA    │ 几ms     │ RAM + RTC  │
│ Deep Sleep     │ ~5μA      │ ~8ms     │ RTC Memory │
└─────────────────────────────────────────────────────┘
```

关键差异：
- nRF52840 的 System ON Idle 只需 **1.5μA**，但 RTC 和 RAM 都保持 → **瞬间唤醒**
- ESP32-C3 要达到类似低功耗需要进 Deep Sleep，但会丢失 RAM → **唤醒慢 + 需要重新初始化**
- BLE 键盘 99.9% 时间在"等待按键"，nRF 在这个状态下几乎不耗电

#### 4. 片上 DC-DC 转换器

```
nRF52840 电源路径:

USB 5V ──→ LDO ──→ 3.3V (给外部)
             │
             └──→ 片上 DC-DC ──→ 1.3V (给 CPU 内核)
                   效率 ~90%

ESP32-C3:
USB 5V ──→ 外部 LDO ──→ 3.3V ──→ 内部 LDO ──→ 1.1V (给 CPU)
                                    效率 ~60-70%
```

nRF52840 内置高效 DC-DC 转换器：
- 3.3V → 1.3V 转换效率约 **90%**
- 而 LDO 的效率只有 1.3/3.3 = **39%**
- 光这一项就省了一半多的电

#### 5. 快速唤醒 (64MHz HFCLK)

nRF52840 内置 64MHz RC 振荡器：
- 从睡眠唤醒到 CPU 运行只需 **<2μs**
- 不需要等外部晶振起振（32MHz 晶振要 ~360μs）
- BLE 的连接事件很短（通常 <1ms），快速唤醒意味着更快干完活、更快睡下去

### 实际键盘场景功耗对比

假设一个 BLE 键盘，连接间隔 30ms（标准 BLE HID）：

```
nRF52840:
  睡眠 29.5ms × 1.5μA  = 44.25 μA·ms
  唤醒 0.5ms × 5mA     = 2500 μA·ms
  平均: ≈ 85μA
  CR2032 纽扣电池 (220mAh): 可用 ~108 天

ESP32-C3:
  睡眠 29ms × 800μA    = 23200 μA·ms
  唤醒 1ms × 50mA      = 50000 μA·ms
  平均: ≈ 2440μA (2.4mA)
  CR2032 纽扣电池 (220mAh): 可用 ~3.7 天
```

**nRF52840 的电池寿命是 ESP32-C3 的约 30 倍**，这就是为什么几乎所有无线键盘都用 Nordic 芯片。

## nRF52840 硬件架构细节

### 存储架构

```
┌─────────────────────────────────────────┐
│              nRF52840 芯片               │
│                                         │
│  ┌─────────┐    ┌──────────────────┐   │
│  │ CPU      │    │ Flash (1MB)      │   │
│  │ M4F      │◄──►│ 0x00000-0xFFFFF  │   │
│  │ 64MHz    │    │ 读: 0 等待周期    │   │
│  │ FPU      │    │ 写: 需要 NVMC    │   │
│  │ MPU      │    └──────────────────┘   │
│  └────┬─────┘                           │
│       │ AHB 总线                         │
│  ┌────┴─────┐    ┌──────────────────┐   │
│  │ RAM      │    │ FICR (只读)       │   │
│  │ 256KB    │    │ 芯片 ID、校准数据  │   │
│  │ 8 个 Bank │    └──────────────────┘   │
│  └──────────┘    ┌──────────────────┐   │
│                  │ UICR (用户配置)    │   │
│  ┌──────────┐    │ Bootloader 地址   │   │
│  │ 系统 ROM  │    │ NFC 引脚配置      │   │
│  │ (无用户   │    └──────────────────┘   │
│  │  代码)    │                           │
│  └──────────┘                           │
└─────────────────────────────────────────┘
```

- **Flash**: 1MB，分页 4KB，内置 NVMC (Non-Volatile Memory Controller) 控制读写
- **RAM**: 256KB，分 8 个 bank，可独立关闭省电
- **FICR**: 工厂写入的只读寄存器，包含芯片唯一 ID、蓝牙地址、温度校准等
- **UICR**: 用户可配置寄存器，存 Bootloader 起始地址等
- **系统 ROM**: 出厂固化的极少代码（不含 bootloader），只有基本硬件初始化

### 外设互连 (PPI + EasyDMA)

```
┌──────────┐   PPI    ┌──────────┐   PPI    ┌──────────┐
│  TIMER   │─────────►│  RADIO   │─────────►│  GPIO    │
└──────────┘          └──────────┘          └──────────┘
                           │
                      EasyDMA
                           │
                      ┌────┴────┐
                      │  RAM    │  (CPU 不参与数据搬运)
                      └─────────┘
```

### 安全特性

- **APPROTECT**: 可以锁定 SWD 调试端口，防止固件被读取
- **硬件 AES-128**: BLE 加密不走 CPU
- **硬件 RNG**: 真随机数生成器，用于密钥生成
- **ARM TrustZone** (nRF5340 有，nRF52840 没有)
